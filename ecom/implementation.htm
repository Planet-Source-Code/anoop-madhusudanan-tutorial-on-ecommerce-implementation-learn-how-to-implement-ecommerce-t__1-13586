 IMPLEMENTATION
1. COMMUNICATION BETWEEN COMPUTERS
Hey, go out and have a Coke if you are getting bored. Better, if you need refreshment, just take a visit to my site and then come back :-). A little more theory now - before going to the Project section, to do what you learned 'practically'.

Before going any step further, you need a basic idea of computer communication. So, I will give you an idea about Network Interfaces and Network Protocols for developing transaction applications. Even if you are getting bored, stay with me.

For understanding how two computers communicate each other, you have to know the terms 'Network Protocols' and 'Network Interfaces', and surely, the difference between them. After all, remember that I am explaining them from a general angle, and not from the perspective of any specific operating system. 

1.1) Network Protocols: 

Now let me explain what is a Network Protocol (or Transport Protocol). A Network Protocol "packages data that is to be sent on the network in a way that the computer on the receiving end can understand." Simply, they are responsible for transferring data over a network.

Now, read the next line, by keeping your shoes in place of your hat - so that you will never forget it in your life: "TCP/IP, UDP/IP, XNS, NetBEUI, and IPX are Transport Protocols. They are used for transferring data over a network.".

In fact, you don't have to worry much about Network Protocols. This is simply because, as a programmer, you are going to access these underlying protocols through Network Interfaces, which are described below.

1.2) Network Interfaces: 

It is very important to understand the difference between Network (Transport) Protocols and Network Interfaces. As I already pointed, network interfaces are used to provide a means for applications (or programs) to access the underlying protocols, which actually 'transfers' the data over a network.

Well, again it is the time to replace your hat with your shoes: Here is the next important point. "Sockets, Net BIOS, Named Pipes and NetDDE are the popular Network Interfaces. These interfaces do not define how the data is actually transferred (which is actually the job of the underlying Network Protocol)". (I am not discussing issues like name resolution/physical address resolution here, because it is beyond the scope of this article. Anyway, I would like to tell you that each interface - say Named Pipes or Net BIOS - has its own method of resolving addresses). You can use any of these interfaces to implement communication facilities in your applications. (Based on your purpose, Operating System and development environment). For example, you can develop networking applications that uses TCP/IP protocol, by using the Socket Interface from Visual Basic - i.e. by using  the WinSock (Windows Sockets) API provided by Microsoft.
2. USING TCP/IP FOR MANAGING TRANSACTIONS
You may wonder - "why this man is discussing all these stupid network stuff, to discuss transactions?". The reason is simple. You need to understand the basic concepts of networking (the facts I described above), to understand the Implementation of ECommerce transactions fully - at least in my opinion.

OK, as you already read, almost any computer in the net knows one common, simple (to them) language - and that is nothing other than TCP/IP. So, computers in the net generally uses TCP/IP - in most cases - to communicate each other. For example, consider the way you browse Planet Source Code. You direct your browser to PSC, it contacts PSC web server, and then sends/gets required information to display. I may write later about the working of web servers, mail servers etc - but now, let me continue by sticking to the subject.
3. IMPLEMENTING SOCKET INTERFACE

Windows developers can have the luxury of using Winsock API in their applications, to manage TCP/IP connections. In Visual Basic, you can use the Microsoft Winsock Control, for implementing TCP/IP facilities in your application. The basic logic of any TCP/IP application or service - including World Wide Web, FTP applications, Chat programs etc - remains the same. 

(I think you are comfortable with  basic socket implementation - at least to write a simple chat program. But if you are not familiar, don't worry. Just download few simple multi user chat programs/chat tutorials from Planet-Source-Code, and learn the implementation well)

The basic implementation of an ECommerce transaction with WinSock is simple. Once you signed up yourself with a 3rd Party Service to simplify your transactions between the acquirer, you can easily act as their client to manage transactions (see the section 'The Concepts' if you are getting confused). As in the case of any ECommerce transaction, the process is so simple

  • Open a socket by using the given IP (most probably, the IP of the 3rd Party Service Provider) and Port (they will give you a port number)
  • Exchange the data (like the credit card number, related control string etc)
  • Get the reply if any
  • Close the socket

Once a socket is opened, the exchange of data is regulated with a given set of protocols. Most third party service providers have their own protocols (they will provide the documentation to you) - to manage these transactions.
